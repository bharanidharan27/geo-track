System Architecture Overview

The GeoTrack platform uses an event-driven pipeline: scan events are ingested by Kafka producers, streamed through regional topics, processed by an idempotent consumer, and stored in a partition-aware CockroachDB cluster. FastAPI exposes REST APIs for tracking and administration, while Prometheus and Grafana close the loop with end-to-end observability.

Design principles

  • Geo-distributed design with workload and data locality across AWS regions.
  • Event-sourced scan history using append-only streams and ordered partitions.
  • Strong consistency and exactly-once semantics at the application layer.
  • Built-in fault tolerance using replication, DLQs, retries, and Raft.
  • First-class observability for latency, throughput, and recovery behavior.

Key tables & partitioning

CockroachDB schema
  • Accounts — REGIONAL BY ROW, keyed by home_region.
  • Parcels — REGIONAL BY ROW, keyed by destination_region.
  • ScanEvents — REGIONAL BY ROW, keyed by facility_region.
  • Carriers — GLOBAL, replicated across all regions.
Cluster guarantees
  • Replication factor RF = 3 with Raft-based consensus for every range.
  • Automatic leaseholder re-election on node or zone failures.
  • Serializable isolation for all OLTP queries and writes.

Major Components

The system is composed of modular services, each focusing on a single responsibility. Together they form a resilient and scalable parcel tracking pipeline.

Ingestion

Kafka Producers & Topics

  • Scan events are pushed into region-specific topics (e.g., aws-us-west-2).
  • Key-based partitioning on tracking_id preserves per-parcel ordering.
  • Producers configured with acks = all to ensure durable commits.
Streaming Backbone

Kafka / Redpanda Cluster

  • Distributed log with multiple partitions per topic for parallelism.
  • Replicated partitions across brokers for durability and availability.
  • Supports consumer groups for scalable consumption.
Processing

Kafka Consumer Service

  • Consumes regional topics in batches and prepares DB-ready payloads.
  • Performs idempotent UPSERTs to achieve exactly-once behavior.
  • Implements DLQ routing, retries, and backoff for failed messages.
Storage

CockroachDB Multi-Region Cluster

  • REGIONAL BY ROW layout to co-locate data with traffic.
  • Raft consensus with RF=3 and quorum writes (2 / 3).
  • Automatic range rebalancing and leader transfers on faults.
API Layer

FastAPI Service

  • Admin APIs: /admin/accounts, /admin/carriers, /admin/parcel, /admin/scan.
  • Tracking API: GET /track/{tracking_id} returns parcel status and history.
  • Stateless services for simple horizontal scaling.
Monitoring

Prometheus & Grafana

  • Prometheus scrapes metrics from Kafka, FastAPI, and CockroachDB.
  • Grafana dashboards visualize latency, throughput, and replication health.
  • Used to observe chaos tests and recovery behavior.

End-to-End Data & Control Flow

A parcel’s journey is modeled as a sequence of scan events. Each event travels through the streaming pipeline to become a strongly consistent row in the database, visible via REST APIs.

1
Event generation

Regional facilities or synthetic generators emit scan events (handoff, arrival, departure, delivered) with timestamps and facility region metadata.

2
Kafka ingestion

Producers publish each event into the appropriate regional Kafka topic. The tracking_id determines the partition, ensuring ordered events per parcel.

3
Durable streaming

Kafka replicates partitions across brokers. With acks=all, an event is acknowledged only after all in-sync replicas persist it.

4
Consumer processing

The consumer service reads batches, deduplicates on composite keys (tracking_id, event_ts, event_type), and executes transactional UPSERTs into CockroachDB.

5
Database storage

CockroachDB stores accounts, parcels, carriers, and scan events with per-table regional keys, enabling local reads and strongly consistent writes under serializable isolation.

6
Client access & monitoring

FastAPI exposes tracking endpoints to the UI. Prometheus scrapes metrics and Grafana visualizes system behavior, including chaos experiments and fault recovery.

Distributed System Characteristics

GeoTrack demonstrates classic distributed database properties in a realistic logistics scenario, highlighting how partitioning, replication, and consistency interact under load and failures.

Partitioning & locality

  • Regional Kafka topics isolate workloads and failures by geography.
  • REGIONAL BY ROW tables route queries and writes to nearby replicas.
  • Tracking queries benefit from locality when run from the same region as the data.

Replication & consistency

  • Kafka replication factor and acks=all provide durable event logs.
  • CockroachDB uses Raft, RF=3, and quorum writes for strong consistency.
  • Serializable isolation guarantees that concurrent transactions behave as if sequential.

Fault tolerance & observability

  • Dead Letter Queues capture malformed or repeatedly failing messages for later replay.
  • Automatic leaseholder re-election and range rebalancing keep the cluster healthy.
  • Chaos tests (network delay, broker restart) are monitored via latency and error metrics.
  • Targets: RTO < 30 seconds, error rate < 2%, zero data loss under tested scenarios.
